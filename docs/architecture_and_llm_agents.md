# Архітектура застосунку та LLM-агенти

Короткий огляд структури Pazuzu та ролі LLM-агентів.

---

## Архітектура застосунку

**Pazuzu** — застосунок для роботи з аукціонами ProZorro.Sale та оголошеннями OLX. Користувач взаємодіє через **Telegram-бота** або **Telegram Mini App**; обробка запитів орієнтована на **LLM-агентів**.

### Шари

1. **Точка входу**  
   - `main.py` → `Application`: запуск у режимі «генерація файлу» (`--generate-file`) або Telegram-бота (за замовчуванням).  
   - Опційно запускаються **MCP-сервери** (`--start-mcp`) для надання інструментів агенту.

2. **Клієнти**  
   - **TelegramBotService** — приймає повідомлення, викликає `MultiAgentService.process_query`, відправляє відповіді та файли.  
   - **Telegram Mini App** (FastAPI) — чат з тим самим `MultiAgentService`, валідація `initData`.

3. **Оркестрація запитів**  
   - **MultiAgentService** — єдина вхідна точка обробки запиту: перевірка безпеки → інтерпретація наміру → або мультиагентний пайплайн (звіти/експорт), або один **LangChain-агент** з інструментами та пам’яттю.

4. **Бізнес-логіка та дані**  
   - **Сервіси**: ProZorro, OLX, експорт, геокодування, планувальник подій, користувачі тощо.  
   - **MCP-сервери** (schema, query_builder, analytics, report, export, geocoding, data_update) надають **інструменти (tools)** агенту; агент не має прямого доступу до БД/API.  
   - **Репозиторії** та **MongoDB** — зберігання аукціонів, оголошень OLX, кешів, логів агентів, планованих подій.

5. **LLM**  
   - **LLMService** — виклики Gemini API, кеш, rate limit.  
   - **LangChainAgentService** — один агент на LangChain з MCP-інструментами, пам’яттю розмови (буфер + саммарі + опційно векторний пошук).  
   - **LLMAgentService** — альтернатива на нативному Gemini (function calling), для сценаріїв без LangChain.

---

## LLM-агенти: призначення та функції

| Агент | Призначення | Функції |
|-------|-------------|--------|
| **SecurityAgent** | Захист від шкідливих/експлойтних запитів | Перевірка патернів (DB, `$regex`, цикли, eval тощо), ліміт довжини запиту. При порушенні — відмова та сповіщення адмінів. |
| **InterpreterAgent** | Розбір запиту в структурований намір | **Двоступенева маршрутизація:** (1) Rule-based fast path — без LLM: явні патерни (звіт/експорт за добу/тиждень, slash-команди, кнопки, Mini App) → пайплайн; (2) лише якщо не збіглося — крок 2 (LLM або fallback intent=query) → LangChain-агент. Поле `routing_path`: `rule_based` | `llm`. Зменшує навантаження на LLM, стабільна поведінка, легший дебаг. |
| **PlannerAgent** | Побудова кроків для звітів/експортів | Формує пайплайн: оновлення даних → вибірки по колекціях (з фільтрами за датою/регіоном) → збереження в тимчасові вибірки → експорт у файли. |
| **AnalystAgent** | Виконання кроків пайплайну | Викликає MCP-інструменти (run_tool): trigger_data_update, save_query_to_temp_collection, export_from_temp_collection, generate_report тощо. |
| **AssistantAgent** | Координація мультиагентного потоку | Збирає контекст, викликає інтерпретатора та планувальника, запускає аналітика по кроках, формує фінальну відповідь (з числами за наявності результатів). Логує намір та дії в `agent_activity_log`. |
| **LangChain-агент** (один агент у LangChainAgentService) | Вільний діалог та складні запити | Один агент з повним набором MCP-інструментів, пам’яттю розмови, системним промптом (глосарій, дата/час, контекст архітектури). Викликає tools у циклі (agent loop) з обмеженням ітерацій; підтримує експорт у файли та відправку їх користувачу. |

### Компонент → LLM / rule-based / deterministic

| Компонент | Тип | Примітка |
|-----------|-----|----------|
| SecurityAgent | rule-based | Regex, патерни, ліміт довжини; без LLM. |
| InterpreterAgent | rule-based + опційно LLM | Маршрутизація: rule-based fast path; крок 2 — fallback або LLM з confidence. |
| PlannerAgent | deterministic | Код формує кроки за схемою (plan_step_schema); без LLM. |
| AnalystAgent | deterministic | Виконує кроки через run_tool; без LLM. |
| AssistantAgent | coordinator | Оркеструє інтерпретатор, планувальник, аналітик; без власного LLM. |
| LangChain-агент | LLM | Один агент з tools та пам'яттю для вільного діалогу. |

### Потік запиту (спрощено)

1. Користувач надсилає повідомлення (бот або Mini App).  
2. **SecurityAgent** перевіряє запит; при порушенні — відповідь «запит не може бути виконаний» та опційно сповіщення адмінам.  
3. **InterpreterAgent** — двоступенева маршрутизація: крок 1 rule-based (без LLM) для явних патернів; при незбігу — крок 2 (LLM/fallback). У результаті поле `routing_path`: `rule_based` або `llm`. Для явних запитів типу «звіт за добу по нежитловій нерухомості» інтерпретатор формує структурований намір з `intent=report_last_day`, `period_days`, `collections`, а також семантичними фільтрами (`region_filter`, `filters.property_type` тощо).  
4. Якщо намір — `report_last_day` / `report_last_week` / `export_data` і вказані колекції → **AssistantAgent** виконує пайплайн (Planner → Analyst): `PlannerAgent` перетворює структурований намір у детермінований список кроків (оновлення даних, вибірки з урахуванням фільтрів за датою/регіоном/типом об’єкта, експорт у файли), `AnalystAgent` послідовно викликає MCP-інструменти згідно плану. Відповідь і файли повертаються користувачу.  
5. Інакше (складний або вільний запит) → **LangChainAgentService.process_query**: один агент з інструментами та пам’яттю обробляє запит у циклі (LLM → tool calls → результати → знову LLM) до фінальної відповіді.  
6. Відповідь та згенеровані Excel-файли передаються клієнту (бот відправляє документи; Mini App — по API).

### Стани запиту (state machine)

Для трасування та дебагу запит проходить стани: **RECEIVED** → **SECURITY_CHECKED** → **INTERPRETED** → **PLANNED** (або **ROUTED_TO_AGENT**) → **EXECUTED** → **DELIVERED**. Стани логуються в `agent_activity_log` (step=state, payload.state).

### Контракти

- **LLM ↔ MCP:** Агент викликає лише зареєстровані tools з аргументами згідно JSON Schema. MCP не приймає `raw_query` та заборонені оператори (див. MCP deny-rules у `docs/llm_agent_audit.md`).
- **MCP ↔ БД:** MCP використовує лише QueryBuilder / репозиторії; без прямого виконання довільних запитів. Дозволені колекції: `prozorro_auctions`, `llm_cache`, `olx_listings`. Заборонені оператори: `$where`, `$eval`, `$function`, `$expr`, `$regex` (з обмеженими винятками), `$text`.
- **Client ↔ Application:** Клієнт надсилає текст або `explicit_intent` + `explicit_params` (кнопки, Mini App). Отримує текст відповіді та список файлів (base64) або `artifact_id`; для підтвердження великого експорту — повідомлення з `confirm_token`, після підтвердження — `confirm_export:<request_id>`.

Детальніше: `docs/llm_agent_audit.md`, `docs/developer_glossary.md`.

## Приклад: логіка обробки запиту «Виведи топ-10 оголошень OLX за останній тиждень по Києву та області з найдорожою нерухомістю»

Нижче описано, як саме обробляється такий запит — від входу до відповіді або досягнення ліміту ітерацій.

### 1. Вхід і безпека

- Запит надходить у **MultiAgentService.process_query** (з Telegram-бота або Mini App).
- **SecurityAgent** перевіряє текст на заборонені патерни (прямі звернення до БД, `$regex`, eval тощо) та довжину. Якщо перевірка не пройдена — відмова та опційно сповіщення адміністраторів.
- Стан: **RECEIVED** → **SECURITY_CHECKED**.

### 2. Інтерпретація наміру (InterpreterAgent)

- **Двоступенева маршрутизація:**
  - **Крок 1 (rule-based):** перевіряються патерни типу «звіт за добу», «експорт за тиждень», slash-команди (`/report_day`, `/export`), кнопки та `explicit_intent` з клієнта. Запит «Виведи топ-10 оголошень олх за останній тиждень…» не збігається з цими патернами (немає точного «звіт за тиждень» / «експорт за тиждень» у потрібній формі).
  - **Крок 2:** викликається **fallback** без LLM: повертається структурований результат з `intent=query`, `confidence=0.5`, `routing_path=llm`.

- У **MultiAgentService** перевіряється умова для мультиагентного пайплайну:  
  `use_multi_agent = (intent in ("report_last_day", "report_last_week", "export_data") and structured.get("collections"))`.  
  Для `intent=query` це **False**, тому мультиагентний пайплайн (AssistantAgent → Planner → Analyst) **не використовується**.

- Стан: **INTERPRETED** → **ROUTED_TO_AGENT**. Далі викликається **LangChainAgentService.process_query**.

### 3. LangChain-агент: цикл «LLM → tools → LLM»

- Формується контекст: системний промпт (правила, схеми, глосарій), контекст дати/часу (діапазони «за останню добу/тиждень/місяць» у UTC), пам’ять розмови (буфер, саммарі), поточний запит користувача.
- Агенту доступні інструменти маршруту **free_form** (зокрема `get_collection_info`, `execute_query`, `execute_aggregation`, `get_distinct_values`, `save_query_to_temp_collection`, `export_from_temp_collection` тощо).
- У циклі (до `llm_agent_max_iterations`, за замовчуванням 10):
  1. **LLM** отримує повідомлення та вирішує: викликати tools чи дати текстову відповідь.
  2. Якщо є **tool_calls** — виконуються відповідні MCP-інструменти (наприклад Query Builder MCP: `execute_aggregation` для колекції `olx_listings`), результат у вигляді **ToolMessage** додається в історію (з опційною підказкою `_agent_hint` на початку).
  3. Якщо **немає** tool_calls і є **непорожній текст** — це вважається фінальною відповіддю, вона повертається користувачу.
  4. Якщо немає ні tool_calls, ні тексту — додається підказка (один раз — «сформуй фінальну відповідь з результатів», при повторі — коротша); цикл продовжується.

### 4. Що робить агент під час обробки цього запиту (типово)

- **Ітерація 1:** LLM часто спочатку не викликає tools (лише «думає») — у логах: «Знайдено викликів tools: 0». Додається підказка про фінальну відповідь.
- **Ітерація 2–3:** Агент викликає **execute_aggregation** з pipeline для `olx_listings`. Можливі помилки:
  - Невірна структура: наприклад `$expr: { $in: ["Київ", "$detail.address_refs.city.name"] }` — у MongoDB `$in` очікує масив другим аргументом, а `detail.address_refs` є масивом об’єктів, тому такий вираз призводить до помилки виконання aggregation.
  - Коректна структура — `$match` з `$or` та `detail.address_refs: { $elemMatch: { "city.name": "Київ" } }` (аналогічно для `region.name`). Якщо при цьому **updated_at** у `$match` передано як ISO-рядки, вони на стороні Query Builder нормалізуються в BSON Date; без нормалізації порівняння дати з рядком може дати 0 результатів.
- **Ітерації 4–7:** Агент може пробувати різні варіанти фільтрів (наприклад тільки `$or` по адресах без фільтра за датою, або з `search_data.location`), викликати **get_distinct_values** для `search_data.location`, `detail.resolved_locations.address_structured.city/region`, щоб побачити реальні значення в базі.
- **Ітерація 8:** Якщо pipeline містить тільки `$or` по регіону/місту (Київ, Київська) без фільтра за `updated_at`, aggregation може повернути 10 записів. У відповідь додається підказка: «Користувач не просив файл — дай текстову відповідь».
- **Ітерації 9–10:** Очікується, що LLM дасть фінальну текстову відповідь на основі результатів. Якщо модель повертає порожній content і знову без tool_calls — додається повторна коротка підказка; при досягненні максимуму ітерацій цикл завершується без текстової відповіді (у логах: «Досягнуто максимум ітерацій»).

### 5. Важливі моменти для коректної відповіді

- **Фільтр за датою (OLX):** у `olx_listings` поле `updated_at` зберігається як BSON Date. У pipeline для `execute_aggregation` значення `updated_at` у `$match` можна передавати як ISO-рядки — Query Builder нормалізує їх у дати перед виконанням.
- **Фільтр за регіоном/містом (OLX):** потрібно використовувати `detail.address_refs` з **$elemMatch**, наприклад `{ "detail.address_refs": { "$elemMatch": { "city.name": "Київ" } } }` та аналогічно для `region.name` («Київська» або «Київська область» залежно від даних у довіднику). Не використовувати `$expr` з `$in` по шляху до поля в масиві — це призводить до помилок виконання.
- **Підказки після результатів:** якщо користувач не просив файл (немає слів «excel», «експорт», «файл» тощо), після успішного `execute_aggregation` у контент ToolMessage додається підказка дати саме текстову відповідь, без виклику export. Це має спонукати LLM відповісти текстом з переліком або підсумком замість повторних викликів tools або порожнього content.

### 6. Підсумок потоку для цього запиту

| Крок | Компонент | Дія |
|------|------------|-----|
| 1 | SecurityAgent | Перевірка запиту — пройдена. |
| 2 | InterpreterAgent | Rule-based не збігся → fallback: intent=query, routing_path=llm. |
| 3 | MultiAgentService | use_multi_agent=False → виклик LangChainAgentService.process_query. |
| 4 | LangChainAgentService | Цикл: LLM → execute_aggregation (різні pipeline) / get_distinct_values → ToolMessage з результатами та підказкою. |
| 5 | Query Builder MCP | Валідація pipeline, нормалізація дат у $match для olx_listings, виконання aggregation. |
| 6 | LangChain-агент | Після отримання 10 результатів — очікується текстова відповідь; при порожньому content — підказки, далі або відповідь, або досягнення max ітерацій. |
