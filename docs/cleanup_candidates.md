# Кандидати на очищення проекту

Документ створено за результатами аналізу коду та історії розробки. Що без сумнівів — можна прибирати; що в переліку «сумнівних» — не видаляти без рішення.

---

## 1. База даних (колекції)

### Підсумок

Усі колекції, які оголошено в репозиторіях і міграціях, **використовуються** і **наповнюються** поточним кодом (пайплайн даних, сервіси, адмінка, агент). Окремих колекцій, які можна однозначно вважати «мертвими», не виявлено.

### Як перевірити свою БД

Запустіть:

```bash
py scripts/check_db_structure.py
```

Скрипт покаже:

- **Відсутні очікувані колекції** — потрібно виконати міграції або ініціалізацію.
- **Колекції в БД поза очікуваним списком** — можливі залишки після видалених фіч або старих міграцій. Їх можна видалити вручну через mongosh/Compass, якщо переконаєтесь, що вони не потрібні.

### Не видаляти

- `_migration_history` — потрібна для `run_migrations.py`.
- Усі колекції з `scripts/check_db_structure.py` (список `EXPECTED_COLLECTIONS`) використовуються поточним кодом.

## 2. Сумнівні механіки (не видаляти без рішення)

З історії розробки та коду:

| Механіка | Опис | Чому сумнівно |
|----------|------|----------------|
| **Регламентне оновлення даних** | Планувальник (scheduled_events) + фоновий цикл оновлення в боті | У історії: «тимчасово вимкнено» — фоновий цикл закоментований, оновлення лише вручну з адмінки. UI планувальника та колекція `scheduled_events` лишаються. Можна або повернути авто-оновлення, або прибрати блок «Планувальник» і код, якщо більше не планується. |
| **Відображення аналітики на оголошеннях** | Вкладка «Аналітика» та індикатор ціни (вигідна/середня/дорога) | У історії: «тимчасово прибрано з інтерфейсу» (SHOW_ANALYTICS_UI = false). Бекенд рахує аналітику, API повертає price_indicator. Блок «Перерахувати аналітику» в адмінці лишається. Якщо UI аналітики не планується повертати — можна розглянути прибирання блоку перерахунку з адмінки та спрощення API. |
| **OLX скрапер (клікер)** | Браузерний скрапер у адмінці, блок «OLX скрапер (клікер) — експеримент» | У історії: «експеримент», «не змінювати основні флоу». Якщо експеримент завершено — можна прибрати пункт меню та API; якщо ні — залишити. |
| **Пайплайн pipeline_templates** | Колекція `pipeline_templates`, PipelineRepository, PipelineBuilderAgent | Використовується DomainCacheService (clear_cache після оновлень). Якщо пайплайн-шаблони більше не використовуються в логіці — можна розглянути видалення; інакше лишати. |

## 3. Що можна прибрати без ризику (за бажанням)

- **Логи в консоль** — якщо ще залишились зайві `print`/`logger.info` у критичних шляхах (наприклад, «LLM: початок обробки…») — їх уже прибрано в гілці; переконайтесь, що подібні логи не дублюються.
- **Порожні або тимчасові скрипти** у `scripts/` або `scripts/temp/` — переглянути вручну і видалити непотрібні.

---

## 4. Рекомендації

1. Перед змінами у БД завжди робіть бекап і запускайте `py scripts/check_db_structure.py`.
2. «Зайві» колекції, які показує скрипт, видаляти лише після перевірки, що вони не використовуються жодним кодом або зовнішніми процесами.
3. Сумнівні механіки — вирішувати по черзі: або повернути використання, або офіційно відключити/прибрати з UI та коду та оновити цей документ.
