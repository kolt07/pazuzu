# Developer glossary

## Terms

- **status**: статус тендера/аукціону в OpenProcurement/ProZorro (наприклад: `active`, `active.tendering`, `active.auction`, `complete`, `cancelled`, `unsuccessful`). У поточній реалізації вибірки залишаємо тільки `active` та `active.*`.
- **оголошення**: у контексті цього проекту = **тендер** (сутність API `/tenders`) або **аукціон** (сутність API `/auctions`). "Список оголошень" = результат list endpoint, "деталі оголошення" = результат `GET /tenders/{id}` або `GET /auctions/{id}`.
- **тендер**: процедура закупівлі через ProZorro API (`/tenders` endpoint).
- **аукціон**: процедура продажу майна/активів через ProZorro.Sale API (`/auctions` endpoint). Використовується для продажу нерухомості та інших активів.
- **OLX region slug**: URL-слаг області для фільтрації пошуку на olx.ua (напр. `vin`, `ko`, `lv`). Короткі 2–3 літерні коди з sitemap. Використовується в шляху `.../prodazha-kommercheskoy-nedvizhimosti/{slug}/` або `.../zemlya/prodazha-zemli/{land_type_slug}/{slug}/`. Маппінг: `config/olx_region_slugs.yaml`.
- **OLX land type slug**: URL-слаг типу землі для фільтрації на olx.ua (напр. `zemlya-zhitlovo-y-susplno-zabudovi`, `zemlya-ozdorovchogo-priznachennya`). Використовується в шляху `.../prodazha-zemli/{slug}/`. За замовчуванням с/г виключено. Маппінг: `config/olx_land_type_slugs.yaml`.
- **канонічний URL OLX**: URL оголошення без query-параметрів. Один і той самий оголошення може мати різні URL (search_reason=promoted vs organic) — у БД зберігається канонічна форма. Утиліта: `utils/olx_url.normalize_olx_listing_url()`.
- **OLX скрапер (клікер)**: експериментальний спосіб збору даних з OLX через браузерну автоматизацію (Playwright).
- **OLX_SCRAPER_USE_BROWSER / olx.use_browser**: єдине джерело для всіх оновлень. При увімкненні Phase 1 OLX (run_olx_update_raw_only) отримує сторінки через BrowserPageFetcher (Playwright); області обробляються послідовно. Застосовується скрізь: точкове оновлення, повне оновлення, регламент, адмінка, MCP. Джерело: config/settings.py (olx_use_browser з env та config.yaml olx.use_browser), run_olx_update_raw_only(use_browser=None → settings → env), source_data_load_service передає use_browser з settings. Замість HTTP-запитів відкривається реальний браузер, відтворюється поведінка користувача (навігація, скрол, відкриття сторінок оголошень); HTML передається в існуючі парсери (parse_listings_page, parse_detail_page). Результати зберігаються в raw_olx_listings і далі обробляються тим самим pipeline (Phase 2). Запуск: адмін-панель Mini App → «Дані та аналітика» → «OLX скрапер (клікер)». Не замінює основний флоу оновлення (run_update, source_data_load_service). Джерело: scripts/olx_scraper/clicker_scraper.py.

- **OLX Partner API**: публічне API OLX Europe для партнерів. Дозволяє публікувати та керувати оголошеннями (adverts), отримувати довідники (категорії, регіони, міста), вести листування з користувачами. Не надає публічного пошуку чужих оголошень — лише оголошення авторизованого користувача. Джерело: `docs/partner_api.yaml`. Детальніше: `docs/olx_partner_api_overview.md`.
- **експорт за ідентифікаторами**: механізм збереження даних у файл, коли на вхід подаються ідентифікатори записів у БД (наприклад, `auction_id` або URL оголошення), опціонально — структура полів у файлі. Сервіс генерує файл у тимчасовій папці та повертає посилання на розташування файлу. Реалізовано в `ExportDataService` та інтегровано в механізм формування звітів (report-mcp, `ReportGenerator`).
- **відправка файлу агентом**: після виклику агентом tool `export_listings_to_file` результат містить `file_base64`; обробник історії (`_extract_excel_files_from_history`) знаходить такі ToolMessage та передає список файлів у Telegram-сервіс, який відправляє їх користувачу через `send_document`. Агент не викликає окремий tool для відправки — достатньо викликати export та коротко повідомити, що файл сформовано та надіслано.
- **регламентне оновлення даних**: фонове оновлення даних із зовнішніх джерел (ProZorro, OLX) за заданим інтервалом під час роботи застосунку. У Pazuzu — кожні N хвилин оновлюються дані за минулу добу; інтервал задається в налаштуваннях (`background_update.interval_minutes` або `BACKGROUND_UPDATE_INTERVAL_MINUTES`). Працює лише при запуску в режимі Telegram бота.
- **шарова архітектура (застосунку)**: організація логіки за шарами. **Найвищий шар** — LLM-агент: приймає запити користувача, планує дії, викликає інструменти, формулює відповіді. **Шар бізнес-логіки** — сервіси та MCP-інструменти: валідація, доступ до даних, формування звітів, оновлення джерел. Агент не має прямого доступу до БД/API/файлів; нові можливості додаються через нові сервіси та інструменти.
- **інструменти агента (три категорії)**: (1) **Ініціація обмінів** — оновлення даних із джерел (trigger_data_update з параметрами); (2) **Вибірка, агрегація та інтелектуальна обробка** — схема, метрики, запити, агрегації, геокодування; (3) **Збереження у файл і відправка користувачу** — export_listings_to_file (стандартна або задана структура), save_query_results_to_excel, generate_report.
- **тимчасова вибірка (агента)**: збережені на сервері результати запиту під унікальним `batch_id` (temp_collection_id) для подальшого експорту в файл. Агент не отримує самі документи в контекст — лише ідентифікатор вибірки та кількість записів. Після виклику `export_from_temp_collection` тимчасові дані видаляються. Колекція БД: `agent_temp_exports`; репозиторій: `AgentTempExportsRepository`.
- **пам'ять розмови (агента)**: контекст історії спілкування з користувачем для LLM-агента. У проєкті реалізовані елементи: (1) **ConversationBufferMemory** — буфер останніх N пар (запит, відповідь); (2) **ConversationSummaryMemory** — саммарі старішої частини діалогу через LLM; (3) **VectorStoreRetrieverMemory** — збереження обмінів у векторному вигляді та пошук релевантних фрагментів за поточним запитом (при наявності embeddings, напр. Gemini). Пам'ять ведеться окремо для кожного користувача (user_id).
- **Telegram Mini App**: веб-застосунок, що відкривається всередині Telegram (WebView) і отримує від клієнта рядок **initData**. Сервер має валідувати initData (HMAC-SHA256 з токеном бота) перед використанням; після валідації можна ідентифікувати користувача (user_id) та надавати API. У Pazuzu Mini App дублює функціональність бота: чат з LLM, звіти, адмін-дії. Реалізація: `telegram_mini_app/` (FastAPI + статика), кнопка меню бота (MenuButtonWebApp) веде на `mini_app_base_url`.
- **initData**: рядок у форматі query string, який Telegram передає Mini App при запуску (наприклад, `user=...&auth_date=...&hash=...`). Містить дані про користувача та підпис (hash). Використовувати на сервері лише після валідації підпису (алгоритм: HMAC-SHA256 з ключем "WebAppData" та токеном бота). Джерело: [Telegram Web Apps](https://core.telegram.org/bots/webapps#validating-data-received-via-the-mini-app).

- **мультиагентна архітектура (Pazuzu)**: організація обробки запиту користувача через кілька спеціалізованих агентів. **Агент-помічник** — спілкування з користувачем, контекст бесіди, виклик інтерпретатора та планувальника, виконання пайплайну, відповідь. **Агент-аналітик даних** — виконання запитів до БД, збереження в тимчасові вибірки, експорт у файли (через інструменти MCP). **Агент-інтерпретатор** — розбір запиту в структурований намір, пояснення метрик, витягування структурованої інформації з неструктурованого тексту (описи аукціонів/оголошень). **Агент безпеки** — верифікація запиту на шкідливі/експлойтні патерни; при виявленні — відмова та сповіщення адміністраторів у Telegram. **Агент-планувальник** — формування пайплайну кроків (оновлення даних, вибірки, експорт) при складних запитах. Лог активності агентів зберігається в колекції `agent_activity_log`.

- **теги (оголошень)**: масив коротких ідентифікаторів (рядків у нижньому регістрі), що характеризують оголошення для фільтрації. Формуються агентом-інтерпретатором при LLM-парсингу опису: призначення (крамниця, аптека, офіс, склад, кафе, ресторан тощо), комунікації (газ, вода, електрика, каналізація, опалення, інтернет), інше (ремонт, паркінг). У OLX зберігаються в `detail.llm.tags`; фільтр — `"detail.llm.tags": {"$in": ["тег1", "тег2"]}`. Список унікальних тегів у колекції — `get_distinct_values(collection, "detail.llm.tags", unwrap_array=True)`.

- **сервіс-планувальник (SchedulerService)**: відповідає за планування подій на майбутнє. Події можуть стосуватися системи загалом або конкретних користувачів. Типи подій: **регламентне оновлення даних** (тільки для адміністраторів), **регламентне формування звітів**, **нагадування** (разові або постійні). Реалізація: APScheduler (cron/once), колекція БД `scheduled_events`, репозиторій `ScheduledEventsRepository`. Запускається разом із Telegram ботом; доставка повідомлень/файлів — через `SchedulerNotifier` (Telegram).

- **регламентне формування звітів**: механізм, коли користувач задає параметри виборки, розклад (cron або разово) та умови доставки і отримує файли або текстові аналітичні дані за розкладом. Наприклад: щодня о 9:00 (пн–пт) — файл зі звітом за попередню добу; щодня о 15:00 — текстове повідомлення про зміну середньої ціни за м² по областях. Параметри зберігаються в події типу `scheduled_report` (payload: `report_request`, `delivery`: file|text, `title`).

- **запланована подія (scheduled event)**: запис у колекції `scheduled_events`: тип (data_update | scheduled_report | reminder), scope (system | user), user_id, розклад (cron або once), payload, is_active. Регламентне оновлення даних можуть створювати лише користувачі з роллю admin.

- **зведена аналітика цін (price analytics)**: агреговані метрики з unified_listings за групами (джерело, тип, область, місто) та періодами (день, тиждень пн-нд, місяць). Метрики: середня ціна, ціна за м², ціна за га (UAH, USD); нормальний розподіл (mean, std, q1–q4). Зберігається в колекції `price_analytics`. Перераховується після оновлення даних з джерел. Сервіс: `PriceAnalyticsService`.

- **індикатор ціни (price indicator)**: показник біля ціни на пошуку та сторінці оголошення: **аномально низька** (< q1−1.5×IQR), **вигідна** (≤ q1), **середня** (q1–q3), **дорога** (> q3 до q3+1.5×IQR), **аномально висока** (> q3+1.5×IQR). Типовий діапазон — межквартильний розмах (IQR). Розраховується по місту за останній місяць; якщо по місту < 3 оголошень — fallback на метрику по області. Джерело: `price_analytics` з period_type=indicator.

- **agent loop (цикл агента)**: повторювана послідовність кроків: отримання відповіді від LLM → виконання викликаних інструментів (tool call) → додавання результатів до контексту → наступний виклик LLM. Обмежується параметром **iteration limit** (наприклад, `llm_agent_max_iterations` у конфігурації), щоб уникнути нескінченного циклу та контролювати витрати.
- **LLM асистент vs парсинг**: два окремі налаштування LLM. **Асистент** (діалог, tool-calling, відповіді користувачу) — за замовчуванням **Gemini** (`llm.assistant.provider` / `llm_assistant_provider`, `llm_assistant_model_name`). **Парсинг** описів оголошень (ProZorro, OLX) — за замовчуванням **локальна модель через Ollama** (`llm.parsing.provider` / `llm_parsing_provider`, `llm_parsing_model_name`). Це дозволяє використовувати Gemini для якісного діалогу та Ollama для масового парсингу без API-лімітів.
- **tool call (виклик інструменту)**: вибір моделлю одного або кількох інструментів (MCP tools) та формування аргументів; виконання відбувається на стороні застосунку, результат повертається в контекст для наступного кроку.
- **iteration limit (ліміт ітерацій)**: максимальна кількість оборотів циклу агента за один запит користувача. При досягненні ліміту агент намагається сформувати фінальну відповідь або повертає повідомлення про обмеження. Налаштовується через `llm.agent.max_iterations` або `LLM_AGENT_MAX_ITERATIONS`.
- **guardrails (обмеження безпеки)**: перевірки перед або під час роботи агента: фільтрація заборонених патернів у запиті (SecurityAgent), обмеження довжини вводу, заборона небезпечних операторів у запитах до БД ($regex, $where тощо). При порушенні — відмова та опційно сповіщення адміністраторів.
- **Security Layer**: сукупність перевірок перед обробкою запиту: **SecurityAgent** (патерни, довжина), **rate limiting** по user_id (N запитів за хвилину), **max complexity** (наприклад, довжина запиту). Налаштування: `security_layer.rate_limit_requests_per_minute`, `security_layer.max_query_complexity_length`.
- **логіка обробки запиту (агента-помічника)**: принципи, а не жорсткий список термінів. Агент отримує запит, за потреби уточнює, аналізує доступні інструменти та дані, творчо обробляє і дає відповідь. Відповідь підкріплюється числовими даними за наявності результатів вибірки.

- **двоступенева маршрутизація (InterpreterAgent)**: спочатку **rule-based fast path** (без LLM) — явні патерни: звіт/експорт за добу/тиждень, slash-команди (/report_day, /report_week, /export), кнопки, Mini App, або параметри `explicit_intent`/`explicit_params` з клієнта. При збігу запит йде в мультиагентний пайплайн, у логі зберігається `routing_path=rule_based`. **Крок 2** — лише коли не очевидно або користувач формулює «людською мовою»: використовується LLM або fallback intent=query, далі обробляє один LangChain-агент; у логах `routing_path=llm`. Це зменшує навантаження на LLM, робить поведінку стабільнішою та полегшує дебаг («чому пішов у пайплайн» — бо збіг rule-based).

- **rule-based fast path**: перший крок двоступеневої маршрутизації; без виклику LLM, лише перевірка ключових слів та slash-команд. Реалізація: `InterpreterAgent.try_rule_based_routing()`.

- **гайдбук агента-інтерпретатора**: документ `docs/interpreter_agent_handbook.md`, який описує формати даних для збереження в БД: формати дат (ISO 8601, BSON Date), мова (українська), структура результату парсингу опису (адреси, площі, теги), збереження та вивід топонімів, числа та одиниці. Використовується при розробці та налаштуванні парсингу та інтерпретатора.

- **тест-агент (agent test runner)**: механізм тестування LLM-помічника. Агент (LLM) генерує 5 тест-кейсів різної складності (від простої виборки до агрегацій), кожен кейс відправляється LLM-помічнику; тест-агент має **прямий доступ до БД** (усі колекції), щоб підрахувати очікувані значення та порівняти з відповіддю помічника та тимчасовими вибірками. Результат: «хід думок» (steps) та короткий звіт по кожному кейсу з виявленням елементів, що не відпрацювали. Запуск: меню Адміністрування → «🧪 Тестування агента». Сервіс: `AgentTestRunnerService`.

- **Answer Composer**: компонент, що перетворює результат виконання (дані, тип запиту, розмір, наявність файлу) у єдиний контракт відповіді для клієнта (Telegram, Mini App). Відповідь формується окремо від агента; логіка: короткий список → текст, таблиця → markdown/список, великий обсяг → файл + summary. Для запитів на кількість використовує response_template з IntentDetector. Сервіс: `business/services/answer_composer_service.py`.

- **FinalAnswerRefinementService**: фінальний крок формування відповіді — LLM аналізує запит з контекстом результатів (кількість, фільтри, діагностика) та чернетку від AnswerComposer, формує цілісну людяну відповідь або пояснює, чому запит не може бути виконаний. Сервіс: `business/services/final_answer_refinement_service.py`.

- **контракт відповіді (answer contract)**: єдиний формат відповіді на клієнт: `{ type, title, items, attachments, summary }`. Типи: list, table, chart, file, text. Детально: `docs/answer_contract.md`.

- **знання про колекції (collection knowledge)**: згуртовані результати автоматичного дослідження даних у колекціях: кількість документів, статистика по полях (min, max, avg для числових; топ повторюваних значень та кардинальність для категоріальних). Зберігаються в колекції `collection_knowledge`; оновлюються профілюванням (вручну або за розкладом події типу `data_profile`). Агент отримує короткий блок «загальні знання про дані» у контексті (LangChain system message, multi-agent context_summary), щоб відповідати з урахуванням реального вмісту колекцій. MCP: ресурс `mongodb://collection-knowledge`, інструмент `get_collection_knowledge(collection_name?)`. Сервіс: `CollectionKnowledgeService`.

- **профілювання даних (data profiling)**: автоматичний збір статистики по колекціях на вибірці документів: для кожного налаштованого поля — числова статистика (min, max, avg, count) або розподіл значень (топ N, кардинальність). Запускається сервісом `CollectionKnowledgeService.run_profiling()`; може виконуватися за розкладом (подія `data_profile` у планувальнику). Конфіг полів для профілювання: `FIELD_PROFILE_CONFIG` у сервісі (prozorro_auctions, olx_listings, llm_cache).

- **токени (аналітика LLM)**: кількість вхідних (input_tokens) та вихідних (output_tokens) токенів при викликах LLM API. Зберігаються в логах (event_type=api_usage, metadata.service=llm) у полях metadata.input_tokens, metadata.output_tokens. Використовуються для підрахунку витрат за прайсингом Gemini та відображення в адмін-панелі замість кількості запитів. Джерело: LogsRepository.sum_llm_tokens_*, config/llm_pricing.py.

- **лог обміну LLM (llm_exchange)**: запис у **окремій колекції llm_exchange_logs** з повним текстом запиту та відповіді LLM, хешами (request_hash, response_hash — SHA256), кількістю токенів (input_tokens, output_tokens) та полем **provider** (gemini, ollama). Поля: timestamp, source, request_id, initiator, request_text, response_text, input_tokens, output_tokens, provider. Логується для викликів і до Gemini, і до Ollama. Репозиторій: `LLMExchangeLogsRepository`; запис через `LoggingService.log_llm_exchange()` з LLMService (parse_auction, extract_intent, generate_text) та LangChainAgentService (summarize, main invoke, fallback). Міграція: 043_llm_exchange_logs_collection.

- **токени на оголошення (LLM)**: оцінка споживання токенів LLM при обробці одного оголошення OLX та пов’язаного ОНМ. Складається з: (1) `parse_auction_description` — парсинг опису (~1257 input, ~171 output токенів); (2) `parse_real_estate_objects` — витягування ОНМ (~571 input, ~114 output). Разом ~2113 токенів на оголошення. Скрипт оцінки: `scripts/llm_usage_analysis.py`.

- **price (unified_listings)**: у зведеній колекції під «ціною» за замовчуванням розуміється **price_uah** (ціна в гривнях). Якщо не вказано інакше, усі фільтри та сортування за `price` мапляться на `price_uah`. SourceFieldMapper, pipeline_executor.

- **дерево фільтрів (filter tree)**: структура пошукових фільтрів на рівні домену: **елементи** (поле, тип порівняння, значення) та **групи** (AND/OR), що можуть містити інші групи та елементи. Корінь — група AND за замовчуванням. Геопошук виноситься в окремий елемент з операторами: в межах (INSIDE), не в межах (NOT_INSIDE), в радіусі (IN_RADIUS). Джерело: `domain/models/filter_models.py` (FilterGroup, FilterElement, GeoFilter, GeoFilterElement).

- **рядок фільтрів (filter string)**: текстовий вигляд дерева фільтрів для редагування користувачем. Формат: `"Активність" = True AND "Дата в джерелі" >= '01.01.2026' AND (geo('Область' INSIDE 'Київська'))`. Серіалізація та парсинг: `domain/services/filter_string_service.py` (filter_group_to_string, filter_string_to_models). Поля виводяться лейблами з `config/search_fields.yaml`.

- **UnifiedSearchService**: доменний сервіс централізованого пошуку по unified_listings. Будує FindQuery з дерева фільтрів (FilterGroup + GeoFilter) або з рядка фільтрів, виконує пошук через CollectionManager, повертає список документів та total. Використовується на сторінці пошуку (POST /api/search/query) та при формуванні звітів. Джерело: `domain/services/unified_search_service.py`.

- **root geo (unified_listings)**: базові географічні поля в корені документа: **region** (область), **oblast_raion** (район області), **city** (місто/населений пункт), **city_district** (район міста). Заповнюються з масиву addresses за логікою: якщо всі адреси збігаються — спільне значення; при суперечках — більшість; при наявності точної адреси (is_complete) — її пріоритет. city_district зʼявляється при геокодуванні повної адреси (з вулицею) у великих містах — з sublocality Google Maps. Джерело: UnifiedListingsService._compute_root_geo_from_addresses.

- **агент-осінтер (AssessorAgent)**: LLM-агент для гео-аналізу придатності нерухомості до виду діяльності (аптека, кафе, клініка тощо). Аналізує оголошення (площа, поверх, тип), геокодує адресу, запитує Google Places API (POI навколо), зіставляє з профілем виду діяльності, формує звіт. Детально: `docs/feature_proposal_geo_assessor.md`.
- **профіль виду діяльності (BusinessProfile)**: критерії для оцінки приміщення: параметри приміщення (площа, поверх, тип) та POI-критерії (типи місць навколо, радіуси, роль — позитивний/негативний). Predefined у конфігу або формується LLM з вільного тексту користувача.

- **правила заповнення price_per_m2 та price_per_ha**: (1) Земельна ділянка без нерухомості — price_per_m2 = NULL, price_per_ha заповнена; (2) Земельна ділянка з нерухомістю — обидва поля при наявності даних; (3) Чисто нерухомість — price_per_m2 заповнена, price_per_ha = NULL. Площі землі зберігаються в м² (**land_area_sqm**); для відображення землі використовуються сотки (с) та ціна за сотку (price_per_ha/100). Реалізація: `utils/price_metrics.compute_price_metrics()`; правило: `.cursor/rules/price-metrics-rules.mdc`.

- **площа землі (одиниці)**: у БД зберігається **land_area_sqm** (м²). Для користувача земельні ділянки відображаються в **сотках** (с): 1 сотка = 100 м². Ціна за одиницю площі землі в UI — **ціна за сотку** (грн/с), обчислюється як price_per_ha_uah/100 (у БД зберігається price_per_ha, 1 га = 100 соток).

- **domain-шар (domain layer)**: шар логіки, що огортає сирі дані з БД у domain-об'єкти (entities). Сутності (`UnifiedListing`, `ProzorroAuction`, `OlxListing`, `ListingAnalytics`, `RealEstateObject`) надають методи отримання властивостей та пов'язаних даних. Колекції (`UnifiedListingCollection`, `ListingAnalyticsCollection`, `RealEstateObjectCollection` тощо) дозволяють ізольовано від БД працювати з масивами (filter, sort_by, to_export_rows). **ListingGateway** — точка доступу до оголошень; **CollectionManager** (UnifiedListingsCollectionManager, ListingAnalyticsCollectionManager, RealEstateObjectsCollectionManager) — для пайплайнів. Сервіси та MCP-інструменти агентів використовують domain-шар.

- **PipelineService**: сервіс формування контексту для створення пайплайнів. Отримує інтент з асистента та попередньо виокремлені дані (query_structure), збирає структуру полів колекцій та кеші значень для контексту агенту створення пайплайнів. Джерело: `business/services/pipeline_service.py`.

- **GeoFilterService (business)**: повний сервіс геофільтрації: створення геофільтрів (місто, область, «не в місті», радіус від координат), геокодування адрес через GeocodingService, побудова метрики відстані до точки. Фільтрація за координатами з радіусом — математично (Haversine). Джерело: `business/services/geo_filter_service.py`.

- **тип оголошення (property_type)**: у unified_listings — «Земельна ділянка», «Комерційна нерухомість», «Земельна ділянка з нерухомістю», «інше». Коли користувач каже «нерухомість» без «земля»/«ділянка» — QueryStructureAgent додає фільтр property_type: ['Комерційна нерухомість', 'Земельна ділянка з нерухомістю'], виключаючи чисту землю.

- **нормалізація топонімів**: перетворення назв з запиту користувача у формат, як у БД. «Волинській області» → «Волинська», «Київська область» → «Київська», «у Києві» → «Київ». Правила як при завантаженні з джерел (UnifiedListingsService). Утиліта: `utils/toponym_normalizer.py`; застосовується в GeoFilterService та QueryStructureAgent.

- **RelativeDateResolver**: детермінований сервіс перетворення відносних періодів у конкретні дати. LLM не має генерувати шаблони {{LAST_WEEK_START_DATE}} — тільки {period: "last_7_days"} або {type: "relative", value: "last_week"}. Джерело: `business/services/relative_date_resolver.py`.

- **date_range (CanonicalQuery)**: опис відносного періоду у запиті. Формат: {period: "last_7_days"} або {type: "relative", value: "last_week"}. Після RelativeDateResolver замінюється на date: {gte, lte} з конкретними ISO-датами в filter_metrics.

- **об'єкт нерухомого майна (ОНМ)**: сутність, що описує земельну ділянку, будівлю або приміщення. Зберігається в колекції `real_estate_objects` з полем `type` (land_plot, building, premises). Може бути пов'язана з одним або кількома оголошеннями через `real_estate_object_refs` у unified_listings. Витягується LLM з опису оголошення; кадастрова інформація доповнюється з cadastral_parcels. Сервіс: `RealEstateObjectsService`.

- **кастомна метрика (аналитика)**: метрика, яку задає агент формулою замість вибору з фіксованого списку. У запиті execute_analytics передається `metric` як об'єкт: `{ name, formula, aggregation?, description? }`. **formula** — вираз над полями документів: шляхи `auction_data.*` та `llm_result.result.*`, оператори +, -, *, /, дужки, числа. Парсер у `utils/analytics_formula.py` перетворює формулу на MongoDB aggregation expression без виконання довільного коду. **aggregation**: avg (за замовчуванням), sum, min, max. Формулу можна хешувати (formula_hash) для кешування. Вбудовані метрики лишаються доступними за назвою (list_metrics).
