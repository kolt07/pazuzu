# План пропозицій: нові фішки для покращення UX Pazuzu

Документ містить пропозиції нових функцій, що можуть покращити користувацький досвід роботи з Pazuzu. Пропозиції згруповані за пріоритетом та складністю реалізації.

---

## 1. Високий пріоритет (швидкий ефект)

### 1.1 Візуалізація графіків (chart)

**Поточний стан:** Контракт відповіді вже має тип `chart`, але клієнт (Mini App, бот) не візуалізує графіки.

**Пропозиція:**
- Для аналітичних відповідей (середня ціна по областях, тренд по місяцях) — відображати інтерактивні графіки.
- Бібліотеки: Chart.js або lightweight альтернатива для Mini App.
- Формат даних: `items` з полями `label`, `value` (або `series` для трендів).
- **Користь:** Користувач одразу бачить візуальну картину замість таблиці цифр.

**Складність:** Середня (інтеграція у Mini App, підтримка тем dark/light).

---

### 1.2 Обране / збережені оголошення (watchlist)

**Поточний стан:** Немає механізму збереження обраних оголошень.

**Пропозиція:**
- Користувач може додавати оголошення в «Обране» (кнопка на картці результату пошуку або в деталях).
- Зберігати в колекції `user_favorites` (user_id, source, source_id, added_at).
- Окрема вкладка «Обране» у Mini App — список збережених з можливістю видалення.
- **Користь:** Не потрібно шукати знову; можна порівняти обране пізніше.

**Складність:** Низька (новий репозиторій, API, UI).

---

### 1.3 Розширений фідбек (коментар до дизлайку)

**Поточний стан:** Є like/dislike; при dislike запускається самодіагностика.

**Пропозиція:**
- Опційне текстове поле «Що не так?» при натисканні dislike.
- Зберігати в `llm_feedback` (поле `user_comment`).
- Використовувати для покращення промптів та виявлення типових помилок.
- **Користь:** Якісніший фідбек для ітерацій над агентом.

**Складність:** Низька (розширити API та UI).

---

### 1.4 Швидкі фільтри (збережені пошуки)

**Поточний стан:** Є шаблони звітів; пошук — одноразовий.

**Пропозиція:**
- «Зберегти поточний пошук» — зберегти комбінацію фільтрів (регіон, місто, тип, ціна) під назвою.
- Список збережених пошуків на вкладці Пошук — один клік для застосування.
- Аналогічно до шаблонів звітів, але для пошуку.
- **Користь:** Повторні пошуки без повторного введення фільтрів.

**Складність:** Середня (новий репозиторій, інтеграція з пошуком).

---

## 2. Середній пріоритет (помітне покращення)

### 2.1 Сповіщення про нові оголошення (price / criteria alerts)

**Поточний стан:** Є регламентні звіти за розкладом; немає персоналізованих алертів.

**Пропозиція:**
- Користувач створює «Алерт»: фільтри (регіон, місто, тип, діапазон ціни) + умова (нові оголошення / зміна ціни).
- SchedulerService: подія типу `listing_alert` — періодична перевірка (наприклад, раз на день).
- При збігу — повідомлення в Telegram (текст + посилання на топ-5 нових).
- **Користь:** Користувач не пропустить цікаві пропозиції.

**Складність:** Середня–висока (новий тип події, логіка порівняння «зміни»).

---

### 2.2 Порівняння оголошень

**Поточний стан:** Результати — список; порівняння вручну.

**Пропозиція:**
- «Порівняти» — вибір 2–4 оголошень з чекбоксів.
- Режим порівняння: таблиця з рядками (ціна, площа, адреса, джерело, посилання) по колонках.
- **Користь:** Швидкий вибір між кількома варіантами.

**Складність:** Середня (UI, збір даних для порівняння).

---

### 2.3 Експорт у PDF

**Поточний стан:** Експорт тільки в Excel.

**Пропозиція:**
- Опція «Експорт у PDF» для звітів та результатів пошуку.
- Формат: компактна таблиця з основними полями + посилання.
- **Користь:** Зручніше для перегляду на мобільному, можливість надіслати без змін.

**Складність:** Середня (бібліотека reportlab або weasyprint).

---

### 2.4 Покращення посилань у відповідях

**Поточний стан:** Посилання в Excel — клікабельні; в чаті — текст.

**Пропозиція:**
- У Telegram: inline-кнопки для топ-3 результатів («Відкрити 1», «Відкрити 2», «Відкрити 3»).
- У Mini App: кнопка «Відкрити в новій вкладці» біля кожного посилання.
- **Користь:** Швидкий перехід до оголошення без копіювання URL.

**Складність:** Низька (Telegram inline buttons, target="_blank" у Mini App).

---

## 3. Довгострокові (стратегічні)

### 3.0 Агент-осінтер для гео-аналізу (geo-assessor)

**Поточний стан:** Є геокодування; немає аналізу оточення (аптеки, лікарні, транспорт) для оцінки придатності приміщення.

**Пропозиція:** Агент-осінтер оцінює придатність оголошення для виду діяльності (аптека, кафе, клініка). Використовує Google Places API для пошуку POI навколо адреси. Профілі видів діяльності — predefined або від користувача.

**Детальний опис:** [feature_proposal_geo_assessor.md](feature_proposal_geo_assessor.md)

**Складність:** Висока (новий агент, Places API, профілі, маршрутизація).

---

### 3.1 Інтерактивна карта

**Поточний стан:** Є геокодування, адреси, regions/cities; немає візуалізації на карті.

**Пропозиція:**
- Режим «Карта» на вкладці Пошук — відображення результатів на карті (Leaflet / OpenStreetMap).
- Фільтри залишаються; при зміні фільтрів оновлюються маркери.
- **Користь:** Візуальне розуміння географії пропозицій.

**Складність:** Висока (координати в unified_listings, інтеграція карти, performance).

---

### 3.2 Рекомендації на основі історії

**Поточний стан:** Пам'ять розмови для агента; немає аналізу поведінки користувача.

**Пропозиція:**
- Аналіз: які регіони/типи/діапазони цін користувач переглядав.
- Блок «Можливо, цікаво» — 3–5 оголошень, що відповідають профілю.
- **Користь:** Персоналізація, зменшення рутини пошуку.

**Складність:** Висока (збір подій, профіль, рекомендаційний алгоритм).

---

### 3.3 Спільний доступ до звітів / пошуків

**Поточний стан:** Звіти та експорти — тільки для власника.

**Пропозиція:**
- «Поділитися посиланням» — генерувати тимчасове посилання (TTL 7 днів) на згенерований звіт або збережений пошук.
- Отримувач відкриває без авторизації (або з обмеженням перегляду).
- **Користь:** Легко передати звіт колегам або клієнтам.

**Складність:** Середня–висока (токени, права доступу, UI).

---

### 3.4 Голосовий ввід для запитів

**Поточний стан:** Тільки текстовий ввід.

**Пропозиція:**
- Кнопка мікрофона в полі вводу — запис голосу → Whisper API (або аналог) → текст у поле.
- **Користь:** Зручніше на мобільному, менше друкувати.

**Складність:** Середня (інтеграція Whisper, UI, дозвіли).

---

## 4. Додаткові ідеї (низький пріоритет)

| Фіча | Опис | Складність |
|------|-----|------------|
| Темна/світла тема | Вже є частково; переконатися, що всі екрани підтримують | Низька |
| Мультимовність | Українська за замовчуванням; опційно англійська для інтерфейсу | Середня |
| Прогрес при довгих запитах | Скелетони / «Завантаження…» з етапами (інтерпретація → пошук → відповідь) | Низька |
| Підказки при першому запуску | Онбординг: «Спробуй запитати: знайди оголошення в Києві» | Низька |
| Статистика для користувача | «Ти переглянув 47 оголошень цього місяця» (опційно) | Низька |

## 5. Рекомендований порядок впровадження

1. **Фаза 1 (1–2 тижні):** 1.2 Обране, 1.3 Розширений фідбек, 2.4 Посилання (inline buttons).
2. **Фаза 2 (2–4 тижні):** 1.1 Графіки, 1.4 Збережені пошуки, 2.2 Порівняння.
3. **Фаза 3 (1–2 місяці):** 2.1 Алерти, 2.3 PDF, 3.4 Голосовий ввід.
4. **Фаза 4 (стратегічні):** 3.1 Карта, 3.2 Рекомендації, 3.3 Спільний доступ.

---

## 6. Примітки

- Усі пропозиції узгоджуються з архітектурою LLM-агентів (MCP tools, без прямого доступу агента до БД).
- Нові колекції — через міграції; нові MCP tools — за потреби.
- Потрібно оновлювати `docs/developer_glossary.md` при додаванні нових термінів.
