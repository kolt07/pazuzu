# Гайдбук для агента-інтерпретатора

Документ описує, у **якому вигляді** очікується інформація для збереження в базу даних та для подальшого виводу. Використовується при парсингу описів (аукціони ProZorro, оголошення OLX), формуванні структурованого наміру та при роботі з топонімами й датами.

---

## 1. Мова

- **Мова даних для збереження та виводу:** українська (кирилиця).
- Топоніми, типи нерухомості, комунікації, теги — українською.
- Геокодування: запит можна подавати українською або місцевою назвою; `formatted_address` у відповіді — українською (параметр `language=uk`).
- Не використовувати латиницю для адрес і топонімів у збережених полях (крім технічних ідентифікаторів на кшталт `place_id`).

---

## 2. Формати дат

| Контекст | Формат збереження | Приклад | Примітки |
|----------|-------------------|---------|----------|
| ProZorro: дати в API/БД | Рядок ISO 8601 (UTC) | `2026-02-08T12:00:00.000Z` | Поля `auction_data.dateModified`, `auction_data.dateCreated` — рядки. У запитах ($match) використовувати порівняння рядків, **не** `{"$date": "..."}`. |
| OLX: оновлення запису | BSON Date | — | Поле `updated_at`. У фільтрах: `$gte` / `$lte` з BSON Date або ISO-рядком (залежно від репозиторія). |
| Кеш, логи | ISO 8601 або BSON Date | — | Уніфіковано в межах колекції. |
| **Вивід користувачу** | За потреби — локалізований текст | «8 лютого 2026», «08.02.2026» | Для звітів і інтерфейсу; у БД зберігати канонічний формат. |

**Правило:** при записах у БД не змішувати типи: у колекціях з ISO-рядками дат не записувати BSON Date в ті самі поля.

---

## 3. Структурований вивід з опису (парсинг для БД)

Результат парсингу опису (аукціон, оголошення) зберігається в `llm_cache` і підставляється в документи як `llm_result.result`. Формат має бути єдиним.

### 3.1 Основні поля

| Поле | Тип | Одиниця/формат | Опис |
|------|-----|----------------|------|
| `cadastral_number` | string | — | Кадастровий номер (наприклад `6320685503:03:000:0202`). Якщо немає — порожній рядок. |
| `building_area_sqm` | number або '' | м² | Площа нерухомості (будівлі, приміщення) у квадратних метрах. Число з крапкою як десятковий роздільник. Якщо тільки земля — порожнє. |
| `land_area_ha` | number або '' | га | Площа земельної ділянки в гектарах. Число з крапкою. Якщо тільки приміщення — порожнє. |
| `property_type` | string | — | Один з: «Земля під будівництво», «Землі с/г призначення», «Нерухомість», «інше». |
| `utilities` | string | — | Підведені комунікації через кому: електрика, вода, газ, опалення. Або «відсутні». |
| `floor` | string | — | Поверх (для приміщень), якщо є. |
| `arrests_info` | string | — | Текст про обтяження (арешти) або «відсутні». |

### 3.2 Адреси (масив `addresses`)

Кожен елемент — об'єкт з полями (всі рядки; якщо немає — порожній рядок):

| Поле | Приклад | Правила |
|------|---------|--------|
| `region` | «Харківська», «Київська» | Назва області **без** слова «область». м. Київ і м. Симферополь **не** в складі областей — для них порожнє або окреме правило. АР Крим — «АР Крим». |
| `district` | «Лозівський», «Бориспільський» | Район області. |
| `settlement_type` | «м.», «с.», «смт.» | Скорочено. |
| `settlement` | «Київ», «Верхньоводяне» | Населений пункт **без** префікса (м., с. тощо), з великої літери. |
| `settlement_district` | «Шевченківський район м. Києва» | За наявності. |
| `street_type` | «вул.», «просп.» | Тип вулиці. |
| `street` | «Незалежності» | Назва вулиці без типу. |
| `building` | «39», «27» | Номер будинку. |
| `building_part` | «корпус А» | Блок/корпус, якщо є. |
| `room` | «кв. 5», «офіс 12» | Приміщення, якщо є. |

Якщо в тексті кілька адрес — витягувати всі в один масив `addresses`. Для зворотної сумісності використовується також «плоский» формат: `address_region`, `address_city` тощо — нормалізатор перетворює їх у один елемент масиву `addresses` з полями `region`, `settlement` тощо.

### 3.3 Теги (`tags`)

- **Тип:** масив рядків.
- **Регістр:** лише **нижній регістр**.
- **Зміст:** призначення (крамниця, аптека, офіс, склад, кафе тощо), комунікації (газ, вода, електрика, каналізація, опалення), інше (ремонт, паркінг). Лише те, що є в тексті; не вигадувати.
- **Унікальність:** без дублікатів.
- У БД (OLX) зберігаються в `detail.llm.tags`; фільтрація — `{"detail.llm.tags": {"$in": ["тег1", "тег2"]}}`.

## 4. Топоніми: збереження та вивід

### 4.1 Збереження в документі (парсинг опису)

- Область: зберегти як у таблиці вище (`region` без «область»).
- Місто/населений пункт: `settlement`, без префікса «м.»/«с.».
- Топоніми типу «сільрада», «міськрада», «районна рада» — витягувати як є в поле, де доречно (наприклад у `settlement` або окремому коментарі за домовленістю).
- Якщо область не вказана в тексті, але визначається за містом/топонімом — доповнити поле `region`.

### 4.2 Геокодування (вивід та кеш)

- **Вхід:** адреса або топонім (рядок), опційно `region="ua"`.
- **Вихід (для збереження в кеш та виводу):**
  - `latitude`, `longitude` — числа;
  - `formatted_address` — рядок **українською** (відповідь API з `language=uk`);
  - `place_id` — ідентифікатор місця;
  - `address_structured` — словник: `region`, `city`, `street`, `street_number`, `postal_code` тощо (відповідь Google у нормалізованому вигляді).
- Кеш: репозиторій `geocode_cache`; ключ — хеш запиту. Не дублювати запити з тим самим текстом.

---

## 5. Числа та одиниці

- **Площі:** тільки числа, десятковий роздільник — **крапка** (наприклад `956.7`, `0.5296`). Не зберігати рядки типу «956,7» або «25 659,90» — нормалізувати до числа перед записом.
- **Валюта:** суми в гривнях (UAH) — число, без символу валюти в полі.
- Конвертації при парсингу: сотки → м² (×100) або га (×0,01); га → м² (×10000); м² землі → га (×0,0001). Сумувати кілька значень площі одного типу (наприклад кілька приміщень — у `building_area_sqm`).

---

## 6. Формат опису (текст від користувача або джерела)

- Вхідний текст опису — у UTF-8, без нормалізації розривів рядків на стороні інтерпретатора (зберігати як є, якщо інше не задано).
- При витягуванні структурованих даних не додавати у поля тексту власних пояснень — лише витягнуті значення або порожні рядки/null за правилами вище.

---

## 7. Де шукати деталі

- **Схема колекцій і полів:** `config/data_dictionary.yaml` — вирівняні поля, типи, джерела (auction_data, llm_cache).
- **Промпт парсингу опису:** `business/services/llm_service.py` — метод `_create_parsing_prompt` у `BaseLLMProvider`; нормалізація результату — `_normalize_result`.
- **Збереження в llm_cache і підстановка в prozorro_auctions:** `business/services/prozorro_service.py` (робота з `llm_result`, `addresses`, `building_area_sqm`, `land_area_ha`).
- **Геокодування:** `business/services/geocoding_service.py`, `data/repositories/geocode_cache_repository.py`.
- **Теги в OLX:** `detail.llm.tags`; парсинг задається тим самим промптом у `llm_service`, збереження — у OLX-пайплайні.

---

## 8. Короткий чеклист для інтерпретатора

- Дати: ISO 8601 рядки для ProZorro; BSON Date для `updated_at` (OLX); при виводі — локалізований формат за потреби.
- Мова: українська для всіх текстових полів і топонімів.
- Адреси: масив `addresses` з полями `region`, `settlement`, `street` тощо; область без «область»; Київ/Симферополь не в складі областей.
- Площі: числа з крапкою; `building_area_sqm` (м²), `land_area_ha` (га).
- Теги: масив рядків у нижньому регістрі, без дублікатів.
- Топоніми при геокодуванні: вивід `formatted_address` українською; кеш за хешем запиту.
