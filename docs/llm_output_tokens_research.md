# Зменшення вихідних токенів LLM: дослідження та практики

## Мета

Зменшити обсяг вихідних токенів без втрати якості відповіді — вихідні токени коштують у 3–8 разів дорожче за вхідні (залежно від провайдера).

## Висновки з тематичних ресурсів

### 1. Пріоритет оптимізації виводу

- Оптимізація **виходу** дає більший ефект за вартістю, ніж скорочення входу.
- Рекомендації: max_tokens / max_output_tokens, чіткі інструкції щодо формату та довжини.

### 2. Конкретні прийоми

| Прийом | Опис |
|--------|------|
| **Обмеження довжини** | `max_tokens` / `max_output_tokens` — кап на довжину відповіді (20–40 % економії). |
| **Чіткі інструкції** | «Поверни тільки JSON», «не більше 3 речень», «список з 5 пунктів». |
| **Few-shot** | Один короткий приклад потрібного формату часто достатній; уникати довгих переліків прикладів. |
| **Stop-послідовності** | Параметр `stop` для зупинки генерації на межі (наприклад, закінчення JSON). |
| **Стислі промпти** | Менше слів у промпті — менше контексту, що «підштовхує» до довших відповідей. |

### 3. JSON vs текст

- Якщо клієнт може розпарсити **короткий структурований текст** (наприклад, ключ: значення по рядку), це може бути менше токенів, ніж повноцінний JSON з дублюванням ключів.
- Для машинної обробки JSON залишається зручним; важливо **не дублювати** інструкцію «Поверни JSON» багато разів у промпті — достатньо однієї чіткої вказівки на початку.

### 4. Політика «запит–контекст–запит»

- Класична тріада «завдання → контекст → закріплення завдання» підвищує і вхідні, і вихідні токени (модель схильна повторювати структуру).
- **Рекомендація:** одна чітка інструкція на початку + контекст; повторення в кінці — лише коли справді потрібне для стабільності (складний формат). Для більшості класифікаторів і парсерів достатньо однієї інструкції.
